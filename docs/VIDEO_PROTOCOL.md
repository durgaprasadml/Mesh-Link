# Video Protocol

The Mesh Link Video Protocol utilizes two dedicated `PacketType` enums:
1. `VIDEO_SIGNAL`
2. `VIDEO_FRAME`

## Video Signal Packets
Used for session negotiation and mid-stream control.

**Payload Format (JSON):**
```json
{
  "type": "INVITE | ACCEPT | REJECT | END | REQUEST_I_FRAME | PAUSE_VIDEO | RESUME_VIDEO",
  "callId": "uuid-string",
  "targetId": "peer-id",
  "codec": "H265",
  "resolution": "R_720P",
  "isScreenShare": false
}
```

## Video Frame Packets
Carries compressed NAL (Network Abstraction Layer) units generated by `MediaCodec`.

**Payload Format (String Split):**
`{sequenceNumber}:{presentationTimeUs}:{Base64EncodedNALUnit}`

- **sequenceNumber**: Long integer to track frame ordering and detect packet loss.
- **presentationTimeUs**: Microsecond timestamp required by the receiving `MediaCodec` decoder to properly pace the video playback.
- **Base64EncodedNALUnit**: The encrypted, Base64-encoded chunk of H.265 video.

## Loss Recovery
Unlike TCP streams, UDP-style mesh packets can drop. If the `VideoStreamManager` detects a sequence number skip (or the decoder throws an error), it issues a `REQUEST_I_FRAME` signal back to the sender. The sender's `VideoCodecManager` instructs the hardware encoder to immediately generate a KeyFrame, instantly recovering the video stream without tearing down the connection.
